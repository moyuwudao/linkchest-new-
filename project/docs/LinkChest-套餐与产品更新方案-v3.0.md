# LinkChest 套餐设计方案 v3.0 & 产品更新方案

> 基于 v2.0 报告与代码实现深度审查后的修正版
> 核心转变：从"限制你能存多少" → "限制系统能承受多少" + "分享获客最大化"
> 日期：2026-05-05

---

## 一、设计原则（四项核心原则）

1. **非服务器瓶颈资源不做限制**，影响服务器运作的原则上均需限制
2. **客户体验强相关的不做套餐限制**，非客户体验强相关的考虑套餐限制
3. **分享是获客核心引擎** —— 分享时长全员开放，分享数量做收敛引导升级
4. **根据用户各等级数量，预估服务器资源**，旗舰版不设"真无限"，保留物理上限防滥用

## 三、关键设计决策说明

### 3.1 为什么收藏/标签/分组不设"真无限"？

技术上 `quota.ts` 的 `limit` 和 `current` 做数值比较，无法表达 `Infinity`。更重要的是：
- DB层面：单个用户若被攻击/脚本滥用，百万级记录会影响索引效率和备份
- 商业惯例：Notion/飞书的"无限"也是"合理使用范围内的无限"
- **999,999 是功能性无限**：正常人类用户一辈子也达不到

### 3.2 为什么分享数量要限制？

分享虽然是快照机制（每次分享创建独立 `share_items` 记录），但：
- 无限分享 = 无限 DB 记录，是隐蔽的存储炸弹
- 分享页性能：每次访问 `/s/:id` 都要查 Share + ShareItems
- **但分享是获客核心**，所以基础版给 10 个（够用但不够爽），是明确的升级引导点

**获客转化路径**：
```
基础用户创建分享 → 被分享者访问 /s/:id
                        ↓
              看到完整内容（grid布局，无密码）
                        ↓
              页脚显示 "用 LinkChest 创建你的收藏集"
                        ↓
              注册后导入该分享（消耗基础版每日导入配额）
                        ↓
              新用户成为基础版 → 开始创建自己的分享
                        ↓
              分享不够用 / 想要密码保护 / 想看谁访问了
                        ↓
              引导升级专业版
```

### 3.3 为什么封面改为"日上传数量"而非"容量限制"？

- 单张封面压缩后约 **20KB~80KB**，体积确实高度可控
- 真正要防的是 **Sharp CPU 瞬时峰值**（压缩是CPU密集型）
- 总量限制的问题：一旦传满永远无法再传，体验极差
- 日限制的好处：今天传满了明天继续，符合日常使用节奏

### 3.4 元数据抓取日上限覆盖哪些场景？

| 场景 | 是否触发元数据抓取 | 说明 |
|------|:--:|:--|
| 手动新增收藏 | ✅ | 用户没填 title 或 coverImage 时触发 |
| 文件导入(CSV/HTML) | ✅ | 导入项缺少 coverImage 或 title 时触发 |
| 分享导入 | ❌ | 直接使用 ShareItem 快照，**最安全** |
| 分享导入后补全 | ⚠️ | 如果导入的收藏缺少封面，后续手动编辑可能触发 |

**关键措施**：文件导入时，如果大量 item 缺少封面，应**批量预检查日配额**，而非全部入队后再慢慢限流。

### 3.5 为什么每日导入要限制"条数"而非"次数"？

- 旧方案 `dailyImportLimit: 3/10/50` 限制的是导入次数
- 单次导入的条数上限由 `maxItemsPerShare` 间接控制（分享导入）
- 真正的冲击点不在"导入次数"，而在"单次导入数量 × 并发用户"
- 改为限制条数更直接：200/1000/5000 条/日

## 五、服务器资源预估（按新方案）

### 5.1 假设用户分布

参考 SaaS 行业典型转化率：

| 用户类型 | 占比 | 人均收藏 | 日均封面上传 | 日元数据抓取 | 日导入条数 |
|----------|:--:|:--:|:--:|:--:|:--:|
| 基础版 | 85% | 80条 | 1-2张 | 5条 | 10条 |
| 专业版 | 12% | 400条 | 5-10张 | 20条 | 50条 |
| 旗舰版 | 3% | 2,000条 | 20-50张 | 50条 | 200条 |

### 5.2 1,000 注册用户场景下的资源估算

| 资源项 | 旧方案估算 | **新方案估算** | 变化 |
|--------|:--|:--|:--|
| **PostgreSQL 数据** | ~300MB | ~400MB | 收藏上限取消，growth更自然 |
| **COS 封面存储** | ~100GB | **~60GB** | 日上传限制使基础用户上传更少 |
| **日均元数据抓取** | ~5,000次 | **~3,000次** | 日上限限制总请求数 |
| **share_items 表** | 增长不可控 | **可控** | 由单次分享容量+分享数量严格限制 |
| **峰值 CPU 压力** | 高 | **中** | 元数据限流+封面日数量限制 |

### 5.3 容量承载结论

新方案下，2核2GB 服务器的理论承载能力：

| 场景 | 安全DAU | 风险DAU |
|------|:--:|:--:|
| 以基础版为主(85%) | **150-200** | 400+ |
| 混合用户 | **120-180** | 350+ |
| 完成配额计数缓存优化后 | **400-600** | 1,000+ |

### 6.4 元数据抓取限流 (`metadata-queue.ts` + `metadata.ts`)

```typescript
// 新增：元数据抓取配额检查
async function checkMetadataQuota(userId: string, batchSize: number): Promise<boolean> {
  const { limits } = await getQuotaUsage(userId)
  
  // 1. 检查并发（正在进行中的抓取数）
  const concurrentKey = `lc:meta:concurrent:${userId}`
  const currentConcurrent = parseInt(await redis.get(concurrentKey) || '0')
  if (currentConcurrent + batchSize > limits.metadataConcurrent) return false
  
  // 2. 检查日上限
  const today = new Date().toISOString().slice(0, 10)
  const dailyKey = `lc:meta:daily:${userId}:${today}`
  const dailyUsed = parseInt(await redis.get(dailyKey) || '0')
  if (dailyUsed + batchSize > limits.metadataDailyLimit) return false
  
  return true
}

// 抓取完成后更新计数器
async function incrementMetadataCounter(userId: string, count: number) {
  const today = new Date().toISOString().slice(0, 10)
  const dailyKey = `lc:meta:daily:${userId}:${today}`
  const concurrentKey = `lc:meta:concurrent:${userId}`
  
  await redis.incrby(dailyKey, count)
  await redis.expire(dailyKey, 86400) // 24h过期
  await redis.decrby(concurrentKey, count)
}
```

### 6.5 封面上传日限制 (`cover.ts`)

```typescript
// 新增：封面上传日配额检查
async function checkCoverDailyQuota(userId: string): Promise<boolean> {
  const { limits } = await getQuotaUsage(userId)
  const today = new Date().toISOString().slice(0, 10)
  const dailyKey = `lc:cover:daily:${userId}:${today}`
  const dailyUsed = parseInt(await redis.get(dailyKey) || '0')
  
  if (dailyUsed >= limits.coverImagesDaily) return false
  
  await redis.incr(dailyKey)
  await redis.expire(dailyKey, 86400)
  return true
}
```

## 八、风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|:--:|:--:|:--|
| 收藏"无限"导致单用户数据量暴增 | 低 | 中 | 999,999防滥用上限；监控异常用户 |
| 分享数量收紧导致基础用户抱怨 | 中 | 中 | 10个分享对轻度用户足够；分享页强调"无限有效期"作为补偿 |
| 元数据限流误伤正常用户 | 低 | 高 | 日上限30条对基础版足够；抓取失败不计数；**超出后自动使用平台默认封面，前端提示"今日元数据抓取已达上限，已使用默认封面，明日自动补全"** |
| 旧用户已超限如何处理 | 中 | 中 | Grandfather原则：已存在数据不删除，仅限制新增 |
| 封面日限制影响用户体验 | 低 | 中 | 5张/日对基础版足够；智能默认封面覆盖80%场景 |
| 导入条数限制影响迁移 | 低 | 高 | 200条/日足以覆盖正常迁移；可分多日导入 |

## 九、产品更新方案（分阶段实施）

### Phase 1：配额配置层（后端配置更新）

**目标**：更新 `config.ts` 中的配额数值，不改动检查逻辑

- [ ] 删除 `coverImages` 字段（数量限制改为日限制）

### Phase 2：配额检查逻辑（后端核心）

**目标**：实现新的配额检查机制

### 关键转变

```diff
- 旧方案：限制"你能存多少书签"（体验杀手）
+ 新方案：限制"系统能为你处理多少封面和元数据"（成本导向）

- 旧方案：基础版 300 收藏 = "试用品"
+ 新方案：基础版无限收藏 = "真正的免费产品"

- 旧方案：付费为了"更大的收藏空间"
+ 新方案：付费为了"更多分享、更快抓取、更长回收站"

- 旧方案：分享限制有效期，限制获客
+ 新方案：分享无限有效期，限制数量引导升级
```

### 预期效果

1. **用户转化率**：基础版体验大幅改善，口碑传播增加，自然流量转化率提升
2. **付费转化逻辑**：从"你不付费就没法用" → "重度用户自然需要更多资源"
3. **服务器稳定性**：元数据限流 + 封面日数量限制，真正保护服务器资源
4. **获客效率**：分享不再限制有效期，每个用户都是长期传播节点
5. **付费引导清晰**：分享数量(10→50→300)和高级功能(密码/统计/封面)是明确的升级理由
