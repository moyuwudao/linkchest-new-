# LinkChest 后台管理方案 v1.0

> ⚠️ **本文档已过时**。当前项目已上线 `/admin` 后台管理功能（用户管理、数据统计、系统配置等），本文档描述的是早期未实现 admin 的状态，仅供参考。

> 基于项目现状（9 张表 / 52 个 API / 3 端应用 / 单服务器部署）设计的后台管理整体方案

---

## 一、现状与痛点

### 1.1 当前缺少的能力

| 场景 | 当前状态 | 问题 |
|------|----------|------|
| 用户增长监控 | 无 | 无法知道每日新增/活跃/留存 |
| 违规内容处置 | 无 | 用户收藏/分享涉黄赌毒，只能连库删 |
| 用户等级调整 | 无 | 需要手动改数据库 `userTier` 字段 |
| 系统公告 | 无 | 无法向全量用户推送维护通知 |
| 数据运营 | 无 | 不知道哪个平台收藏最多、哪个分享最热门 |
| 客服支持 | 无 | 用户反馈问题时无法查看其数据全貌 |

### 1.2 现有数据规模参考

- 9 张核心表，关系清晰，Prisma ORM 维护
- 用户表 `users` 已有 `userTier`（medium/heavy/super）字段
- 分享表 `shares` 已有 `isActive` 开关，无审核字段
- 收藏表 `collections` 无审核/举报/状态字段
- **目前无 admin 角色、无权限系统、无操作日志表**

## 二、后台管理范围：需要管理什么

按优先级分为 **P0（必须）/ P1（重要）/ P2（锦上添花）**。

### 2.1 P0 — 核心运营能力

#### ① 用户管理
| 功能 | 说明 | 涉及表 |
|------|------|--------|
| 用户列表 | 分页、搜索（邮箱/用户名/ID）、筛选（等级/注册时间/状态） | users |
| 用户详情 | 查看该用户的收藏数/标签数/分组数/分享数/最后登录 | users + 各关联表 COUNT |
| 等级调整 | 修改 `userTier`（medium/heavy/super） | users |
| 账号管控 | 禁用/启用账号（软删除标记） | users（需新增 `status` 字段） |
| 登录记录 | 查看用户最近登录时间/IP（现有 `lastLoginAt`） | users |

#### ② 内容审核（收藏 + 分享）
| 功能 | 说明 | 涉及表 |
|------|------|--------|
| 收藏列表 | 查看全量收藏，按平台/用户/时间筛选 | collections |
| 分享列表 | 查看全量分享，按类型/状态/用户筛选 | shares |
| 禁封分享 | 关闭 `isActive = false`，分享链接失效 | shares |
| 违规标记 | 标记违规内容（需新增 `status` / `reportCount` 字段） | collections / shares |

#### ③ 系统看板
| 功能 | 说明 | 数据来源 |
|------|------|----------|
| 今日概况 | 新增用户、新增收藏、新增分享、活跃用户 | 各表 COUNT + 时间筛选 |
| 趋势图表 | 7/30 日注册用户趋势、收藏增长趋势 | 聚合查询 |
| 平台分布 | 全站收藏的平台分布 Top 10 | collections groupBy platform |
| 用户等级分布 | medium / heavy / super 占比 | users groupBy userTier |

### 2.2 P1 — 运营效率工具

#### ④ 系统配置
| 功能 | 说明 | 建议实现 |
|------|------|----------|
| 配额配置 | 动态调整各等级的配额上限 | 当前硬编码在 `config.ts`，改数据库表 |
| 平台配置 | 增删改平台定义（名称/品类/颜色） | 现有 `SUPPORTED_PLATFORMS` 是静态数组 |
| 公告管理 | 全站弹窗/横幅公告（Web + Mobile 展示） | 新增 `announcements` 表 |
| 维护模式 | 一键开启维护，仅管理员可登录 | 环境变量或配置表 |

#### ⑤ 客服支持
| 功能 | 说明 |
|------|------|
| 用户数据导出 | 按用户 ID 导出其全部收藏/标签/分组（CSV） |
| 数据修复 | 批量修改/删除某用户的异常数据 |
| 操作日志 | 记录管理员的所有修改操作（谁/何时/改了什么） |

### 2.3 P2 — 高阶能力

| 功能 | 说明 | 备注 |
|------|------|------|
| 邮件群发 | 向指定用户群发送邮件（SES 已有） | 注意频率，避免进垃圾箱 |
| 数据报表导出 | 全站运营数据 Excel 导出 | 数据量大时走异步任务 |
| 内容推荐干预 | 手动置顶/下架广场内容 | 广场已下线，恢复时可用 |
| 自动审核策略 | 关键词过滤 + AI 内容审核 | 需接入腾讯云/阿里云服务 |

## 三、技术方案

### 3.1 方案对比

| 维度 | 方案 A（推荐） | 方案 B | 方案 C |
|------|---------------|--------|--------|
| **描述** | 在现有 Web 项目新增 `/admin` 路由 | 独立部署一个 Admin 前端 | 引入开源 Admin 框架（Refine） |
| **技术栈** | Next.js 14 + shadcn/ui + TanStack Query | Vite + React + shadcn/ui | Refine + Ant Design |
| **复用程度** | 高（复用现有 API 封装、组件、认证） | 中（复用 API 封装） | 低（需重新适配） |
| **开发成本** | ~40-50 工时 | ~60-70 工时 | ~50-60 工时 |
| **维护成本** | 低（一套代码库） | 中（两套前端需同步） | 中（额外依赖） |
| **部署复杂度** | 极低（随 Web 端一起构建） | 中（需额外域名/端口） | 低 |
| **权限隔离** | 中（前端路由拦截 + API 中间件） | 高（完全独立） | 中 |

**推荐方案 A**：在现有 `apps/web` 中新增 `(admin)` 路由组，复用全部现有基础设施。

### 3.3 数据库变更（最小化）

新增/修改内容：

```prisma
// 1. 用户表新增状态字段
model User {
  // ... 现有字段
  status    UserStatus @default(ACTIVE)  // ACTIVE | SUSPENDED | BANNED
  role      UserRole   @default(USER)    // USER | ADMIN | SUPER_ADMIN
}

enum UserStatus {
  ACTIVE
  SUSPENDED    // 临时封禁
  BANNED       // 永久封禁
}

enum UserRole {
  USER
  ADMIN
  SUPER_ADMIN
}

// 2. 新增管理员操作日志表
model AdminLog {
  id          String   @id @default(uuid())
  adminId     String   // 操作者
  action      String   // 动作类型：BAN_USER / UPDATE_TIER / DELETE_SHARE / ...
  targetType  String   // 目标类型：user / collection / share / config
  targetId    String?  // 目标 ID
  oldValue    Json?    // 修改前的值
  newValue    Json?    // 修改后的值
  ip          String?  // 操作者 IP
  userAgent   String?
  createdAt   DateTime @default(now())

  @@index([adminId])
  @@index([action])
  @@index([createdAt])
  @@map("admin_logs")
}

// 3. 新增系统配置表（替代部分硬编码）
model SystemConfig {
  id        String   @id @default(uuid())
  key       String   @unique  // quota_medium_collections, maintenance_mode 等
  value     Json               // 灵活存储各种类型
  updatedAt DateTime @updatedAt
  updatedBy String?  // 最后修改的管理员

  @@map("system_configs")
}

// 4. 新增公告表
model Announcement {
  id        String   @id @default(uuid())
  title     String
  content   String
  type      String   @default("info")  // info / warning / maintenance
  platforms String[] @default(["web", "mobile"])  // 展示平台
  startAt   DateTime?  // 开始展示时间
  endAt     DateTime?  // 结束展示时间
  isActive  Boolean  @default(true)
  createdAt DateTime @default(now())

  @@index([isActive, startAt, endAt])
  @@map("announcements")
}

// 5. 收藏/分享增加状态（可选，如需审核流程）
model Collection {
  // ... 现有字段
  status    ContentStatus @default(ACTIVE)  // ACTIVE | FLAGGED | REMOVED
}

enum ContentStatus {
  ACTIVE
  FLAGGED      // 被举报/标记
  REMOVED      // 已删除/下架
}
```

### 3.4 API 设计（管理端专用路由）

新增 `apps/api/src/routes/admin.ts`，所有路由前缀 `/api/admin/*`。

| 路由 | 方法 | 功能 | 权限 |
|------|------|------|------|
| `/admin/auth/login` | POST | 管理员登录（独立密码，非用户 JWT） | 无需 |
| `/admin/auth/me` | GET | 当前管理员信息 | ADMIN+ |
| `/admin/dashboard` | GET | Dashboard 核心指标聚合 | ADMIN+ |
| `/admin/users` | GET | 用户列表（分页/搜索/筛选） | ADMIN+ |
| `/admin/users/:id` | GET | 用户详情 | ADMIN+ |
| `/admin/users/:id` | PATCH | 修改用户信息/等级/状态 | ADMIN+ |
| `/admin/collections` | GET | 全站收藏列表 | ADMIN+ |
| `/admin/collections/:id` | DELETE | 删除违规收藏 | ADMIN+ |
| `/admin/shares` | GET | 全站分享列表 | ADMIN+ |
| `/admin/shares/:id/toggle` | PUT | 启停分享 | ADMIN+ |
| `/admin/configs` | GET | 系统配置列表 | SUPER_ADMIN |
| `/admin/configs/:key` | PUT | 修改配置 | SUPER_ADMIN |
| `/admin/announcements` | CRUD | 公告管理 | ADMIN+ |
| `/admin/logs` | GET | 操作日志查询 | SUPER_ADMIN |

### 3.5 认证与权限

```
┌──────────────────────────────────────────────────────────────┐
│                      权限层级                                 │
├──────────────────────────────────────────────────────────────┤
│ SUPER_ADMIN  │ 所有权限 + 配置修改 + 查看操作日志             │
│ ADMIN        │ 查看数据 + 用户管理 + 内容审核 + 公告管理       │
│ USER         │ 只能访问 /api/* 用户端 API                    │
└──────────────────────────────────────────────────────────────┘
```

**实现要点**：
1. 管理员登录使用独立路由，返回带 `role` 声明的 JWT
2. 新增 `requireAdmin` 中间件，检查 `req.user.role >= ADMIN`
3. 新增 `requireSuperAdmin` 中间件，检查 `req.user.role === SUPER_ADMIN`
4. 管理端前端 `/admin/*` 路由增加前端守卫，非管理员跳 403

## 四、对现有服务的压力分析

### 4.1 数据库压力

| 场景 | 压力等级 | 说明 | 缓解措施 |
|------|----------|------|----------|
| Dashboard 聚合查询 | **中高** | `COUNT(*)` + `groupBy` + 时间范围，全表扫描 | Redis 缓存 5 分钟；夜间预计算 |
| 用户列表分页查询 | **中** | 用户量大会触发 `OFFSET` 性能下降 | 游标分页；搜索走索引（email/username） |
| 全站收藏列表 | **高** | 数据量最大的一张表，无用户隔离 | 严格限制返回字段；必须分页；禁止无筛选导出 |
| 详情页关联查询 | **低** | 单用户聚合 COUNT，已有索引覆盖 | — |
| 配置/公告查询 | **极低** | 数据量极小 | Redis 缓存 |

**关键建议**：
- 管理端 API 增加**查询耗时日志**（> 1s 报警），便于及时发现慢查询
- Dashboard 数据走**预计算 + 缓存**，不要每次实时聚合
- 如需大规模导出，改为**异步任务**（生成 CSV 后邮件通知）

### 4.2 Redis 压力

| 场景 | 影响 | 措施 |
|------|------|------|
| Dashboard 缓存 | 少量新增 key：`admin:dashboard:*` | TTL 设 5 分钟 |
| 管理员会话 | 可选：管理员 JWT 黑名单/过期管理 | 复用现有 Redis |
| 限流 | 管理端接口同样需限流 | 复用现有限流中间件 |

**结论**：Redis 压力可忽略，现有实例完全可承载。

### 4.3 API 服务压力

| 场景 | 影响 | 措施 |
|------|------|------|
| 并发请求 | 管理端用户极少（通常 < 5 人同时在线） | 几乎无压力 |
| 大查询阻塞 | 复杂聚合可能阻塞连接池 | Prisma 连接池已配置，复杂查询加 `timeout` |
| 前端构建 | 新增 admin 页面增加构建体积 | 动态导入 + 代码分割 |

**结论**：API 服务压力主要来自**慢 SQL**，而非并发量。

### 4.4 COS / 外部服务压力

管理端几乎不涉及文件上传，仅可能涉及：
- 用户头像查看（读 COS）— 无新增压力
- 批量导出数据 — 无需 COS

## 五、对运营团队的压力分析

### 5.1 人力投入估算

| 职责 | 日常工作量 | 所需技能 | 备注 |
|------|-----------|----------|------|
| **内容审核** | 取决于用户量和举报量 | 无 | 日活 < 1 万时：日均 < 30 分钟 |
| **用户申诉处理** | 被禁用户邮件申诉 | 基础客服能力 | 预计每周 < 10 封 |
| **数据观察** | 每日看板 5 分钟 | 基础数据敏感 | 可培养 |
| **公告发布** | 有维护/活动时才发 | 无 | 月均 1-2 次 |
| **等级调整** | 付费用户/活动赠送 | 无 | 按需处理 |

### 5.2 运营团队配置建议

| 阶段 | 日活 | 建议配置 |
|------|------|----------|
| 冷启动期 | < 1,000 | 开发兼运营，无需专职 |
| 成长期 | 1,000 - 10,000 | 兼职运营 1 人（内容审核 + 客服） |
| 扩张期 | 10,000 - 50,000 | 专职运营 1 人 + 客服 1 人 |
| 成熟期 | > 50,000 | 运营 2 人 + 审核团队 |

### 5.3 降低运营压力的措施

1. **自动化优先**
   - 用户注册邮箱自动验证（已有）
   - 收藏内容关键词自动过滤（敏感词库）
   - 分享链接自动检测（对接腾讯云/百度内容审核 API）

2. **用户自治**
   - 提供举报按钮（收藏/分享），减少运营主动巡查
   - 被举报 N 次自动下架，运营只处理申诉

3. **工具提效**
   - 批量操作（批量禁用分享、批量调整等级）
   - 快捷操作（一键封禁 + 发送通知邮件）

## 七、风险与应对

| 风险 | 概率 | 影响 | 应对 |

## 九、总结

| 维度 | 结论 |
