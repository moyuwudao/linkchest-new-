
# 要求好友共享功能方案探讨

&gt; 文档版本：v1.0
&gt; 创建日期：2026-05-17

---

## 核心需求描述

用户可以向好友发起共享请求，请求对方分享内容。

---

## 方案 A：单向请求模式（推荐）

### 核心思路

用户 A 向用户 B 发起共享请求，B 可接受/拒绝，接受后 B 创建分享并关联到请求。

### 数据库设计

```prisma
model ShareRequest {
  id          String   @id @default(uuid())
  requesterId String   // 请求发起者
  targetId    String   // 请求目标用户
  status      String   // pending | accepted | rejected | expired
  message     String?  // 请求留言
  shareId     String?  // 接受后关联的分享ID
  expiresAt   DateTime // 请求有效期（如7天）
  createdAt   DateTime @default(now())
  
  requester   User     @relation(fields: [requesterId], references: [id])
  target      User     @relation(fields: [targetId], references: [id])
  share       Share?   @relation(fields: [shareId], references: [id])
  
  @@unique([requesterId, targetId])
}
```

### API 设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/share-requests` | POST | 发起共享请求 |
| `/share-requests` | GET | 获取我的请求/收到的请求 |
| `/share-requests/:id/accept` | PUT | 接受请求（自动创建分享） |
| `/share-requests/:id/reject` | PUT | 拒绝请求 |
| `/share-requests/:id` | DELETE | 取消请求 |

### 优点

- 清晰的请求状态管理（pending/accepted/rejected）
- 支持过期机制，自动清理无效请求
- 与现有分享系统完全解耦
- 可扩展支持留言功能

### 缺点

- 需要新增数据库表和 API 路由
- 请求者需要知道目标用户的标识（手机号/邮箱）

---

## 方案 B：基于分享链接的请求模式

### 核心思路

用户生成一个"请求分享"的特殊链接，发送给好友，好友打开后可创建分享并自动关联。

### 实现方式

- 在 `Share` 表增加 `requestedBy` 字段（可选，记录请求来源）
- 新增特殊分享类型 `REQUEST`，用于标识这是一个请求链接
- 好友打开链接后，系统引导创建分享并关联请求者

### 优点

- 复用现有分享基础设施
- 无需知道对方用户标识，通过链接分享

### 缺点

- 分享表结构稍复杂
- 请求状态追踪不够清晰

---

## 方案 C：双向好友关系模式

### 核心思路

建立好友关系，好友间可直接查看对方分享或请求共享。

### 数据库设计

```prisma
model Friendship {
  id        String   @id @default(uuid())
  userId    String   // 用户A
  friendId  String   // 用户B
  status    String   // pending | accepted | blocked
  createdAt DateTime @default(now())
  
  user      User     @relation(fields: [userId], references: [id])
  friend    User     @relation(fields: [friendId], references: [id])
  
  @@unique([userId, friendId])
}
```

### 优点

- 支持双向关系管理
- 可扩展好友动态、消息等功能

### 缺点

- 功能范围较大，复杂度高
- 需要维护好友关系状态

---

## 方案对比总结

| 维度 | 方案A（单向请求） | 方案B（链接请求） | 方案C（好友关系） |
|------|------------------|------------------|------------------|
| **复杂度** | 低 | 中 | 高 |
| **扩展性** | 中 | 低 | 高 |
| **用户体验** | 清晰 | 便捷 | 完整 |
| **开发成本** | 小 | 中 | 大 |
| **推荐度** | ★★★★★ | ★★★☆☆ | ★★★☆☆ |

---

## 推荐方案：方案 A（单向请求模式）

### 理由

1. **聚焦核心需求**：只解决"请求共享"这一个问题，不引入额外复杂度
2. **状态清晰**：pending/accepted/rejected/expired 四种状态覆盖所有场景
3. **独立设计**：与现有分享系统解耦，不影响现有功能
4. **轻量实现**：仅需一张新表和少量 API 路由
