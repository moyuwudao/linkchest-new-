# LinkChest 套餐设计方案重分析报告 v2.0

> 基于《全功能多维评估报告》重新设计，遵循四项核心原则：
> 1. 非服务器瓶颈资源不做限制，影响服务器运作的原则上均需限制
> 2. 客户体验强相关的不做套餐限制，非客户体验强相关的考虑套餐限制
> 3. 根据用户各等级数量，初步预估服务器资源
> 4. 参考其他产品套餐设计方案

---

## 一、当前方案问题诊断

### 1.1 现有配额配置

| 维度 | 基础版(medium) | 专业版(heavy) | 旗舰版(super) | 体验权重 | 服务器压力 |
|------|:--:|:--:|:--:|:--:|:--:|
| 收藏数量 | 300 | 1,000 | 10,000 | 🔴极高 | 🟢低 |
| 标签数量 | 15 | 50 | 100 | 🟠高 | 🟢低 |
| 分组数量 | 30 | 100 | 200 | 🟠高 | 🟢低 |
| 分享数量 | 20 | 50 | 200 | 🟠高 | 🟡中 |
| 分享快照 | 20 | 60 | 200 | 🟡中 | 🔴高(存储炸弹) |
| 上传封面 | 300 | 1,000 | 3,000 | 🟠高 | 🔴高(CPU+存储) |
| 单次分享容量 | 30 | 100 | 500 | 🟡中 | 🔴高(带宽+存储) |
| 每日导入限制 | 3 | 10 | 50 | 🟡中 | 🔴高(CPU+DB) |

### 1.2 核心问题

**问题一：用"数量上限"限制低成本的DB记录，却未限制真正烧钱的资源**

- 收藏、标签、分组本质是纯DB行记录，有索引覆盖，存储极小（单条约0.5KB），查询极轻
- 真正的成本在：**封面图片处理(CPU)**、**元数据抓取(网络+CPU)**、**分享传播(带宽+存储)**、**批量导入(CPU+DB)**
- 现状导致：用户感知被"收藏上限"卡脖子（体验极差），但服务器真正的资源黑洞没有任何限制

**问题二：旗舰版 10,000 收藏 × 3,000 封面的配比不合理**

- 用户如果有 10,000 条收藏，按产品逻辑不可能只上传 3,000 张封面（智能封面覆盖 80% 场景）
- 实际上封面上限应该与收藏上限**解耦**，封面是**增值服务**而非**基础功能**

**问题三：分享是增长引擎却被过度限制**

- 基础版仅 20 个分享 + 30 条/分享，对获客极其不利
- 分享页面带品牌曝光，限制分享等于限制增长
- 应限制**分享带宽消耗**而非分享数量

## 二、重新评估：每个维度的"资源-体验"矩阵

### 2.1 评估框架

```
                    服务器压力高
                         ↑
    【限制区】          【核心限制区】
    非体验+高成本        体验+高成本 → 寻找替代限制方式
    (如: 封面上传数)      (如: 元数据抓取频率)
                         
体验权重低 ←——————→ 体验权重高

    【自由区】          【保护性开放区】
    非体验+低成本        体验+低成本 → 不做限制或设极高上限
    (如: 统计模块)       (如: 收藏/标签/分组数量)
                         
                    服务器压力低
```

### 2.2 全维度重新评级

| 维度 | 服务器压力 | 体验权重 | 当前限制方式 | 新方案建议 | 理由 |
|------|:--:|:--:|:--:|:--|:--|
| **收藏数量** | 🟢低 | 🔴极高 | 硬上限 | **极高上限或不限制** | DB压力可控，是产品核心价值。限制=摧毁体验 |
| **标签数量** | 🟢低 | 🟠高 | 硬上限 | **极高上限或不限制** | 单用户最多几百条，索引查询极轻 |
| **分组数量** | 🟢低 | 🟠高 | 硬上限 | **极高上限或不限制** | 自关联+depth限制已防滥用，查询极轻 |
| **回收站保留** | 🟡中 | 🟠高 | 固定30天 | **按套餐分级: 7/30/90天** | 存储成本随天数线性增长，但影响安全感 |
| **元数据抓取** | 🔴高 | 🔴极高 | 无限制 | **按套餐分级并发/频率** | 最大CPU+网络黑洞，但体验核心→限制频率而非禁用 |
| **封面上传** | 🔴高 | 🟠高 | 硬上限 | **按容量(MB)限制** | CPU密集型+COS存储成本。用户感知容量比数量更直观 |
| **分享数量** | 🟡中 | 🟠高 | 硬上限 | **开放或设高上限** | 获客核心，限制分享=限制增长。改限制分享UV/带宽 |
| **单次分享容量** | 🔴高 | 🟡中 | 硬上限 | **保留限制，调高下限** | 直接关联 share_items 存储炸弹 |
| **每日导入** | 🔴高 | 🟡中 | 硬上限 | **保留限制，调高下限** | 批量CPU+DB冲击，但 3条/日过于苛刻 |
| **分享密码** | 🟢低 | 🟡中 | 全开放 | **全开放** | 零服务器成本，功能开关型限制已被用户厌恶 |
| **深色模式** | 🟢零 | 🟡中 | 全开放 | **全开放** | 纯客户端 |
| **国际化** | 🟢低 | 🟡中 | 全开放 | **全开放** | 无增量成本 |
| **统计模块** | 🟢低 | 🟡中 | 全开放 | **全开放** | 纯聚合查询，无额外成本 |
| **PWA/离线** | 🟢低 | 🟠高 | 全开放 | **全开放** | 纯客户端能力 |

## 三、新套餐设计方案

### 3.1 设计哲学转变

**从"限制你能存多少" → "限制系统能承受多少"**

| 旧思维 | 新思维 |
|--------|--------|
| 收藏300条 vs 1000条 vs 10000条 | 无限收藏，但限制**封面存储空间**和**元数据抓取速率** |
| 分享20个 vs 50个 vs 200个 | 无限分享，但限制**单次分享容量**和**分享页带宽** |
| 标签15个 vs 50个 vs 100个 | 无限标签 |
| 分组30个 vs 100个 vs 200个 | 无限分组（保持3级嵌套不变） |

### 3.2 新配额结构（三级体系）

将配额分为三类：
- **开放类**：不做限制，或设一个极高的防滥用上限（如 999,999）
- **分级类**：按套餐阶梯限制，与服务器成本直接挂钩
- **增值类**：与付费功能直接相关

#### 开放类（所有套餐一致）

| 维度 | 基础版 | 专业版 | 旗舰版 | 说明 |
|------|:--:|:--:|:--|:--|
| 收藏数量 | **无限制** | 无限制 | 无限制 | DB有索引，查询轻。设极高防滥用上限 100,000 |
| 标签数量 | **无限制** | 无限制 | 无限制 | 极高上限 500 |
| 分组数量 | **无限制** | 无限制 | 无限制 | 极高上限 500，保持3级嵌套 |
| 分享数量 | **无限制** | 无限制 | 无限制 | 极高上限 500 |
| 深色模式 | ✅ | ✅ | ✅ | 纯客户端 |
| 国际化 | ✅ | ✅ | ✅ | 无增量成本 |
| 分享密码 | ✅ | ✅ | ✅ | 零成本功能开关 |
| 回收站恢复 | ✅ | ✅ | ✅ | 基础数据安全 |
| 统计模块 | ✅ | ✅ | ✅ | 纯聚合查询 |

#### 分级类（与服务器成本挂钩）

| 维度 | 基础版 | 专业版 | 旗舰版 | 成本关联 | 说明 |
|------|:--:|:--:|:--|:--|:--|
| **封面存储空间** | 50MB | 500MB | 5GB | 🔴COS存储+CPU处理 | 按容量限制比按数量更合理 |
| **单次分享容量** | 50条 | 200条 | 1,000条 | 🔴share_items存储 | 防止分享快照表爆炸 |
| **每日导入上限** | 20条 | 100条 | 无限制 | 🔴CPU+DB批量冲击 | 原3条过于苛刻，打击迁移意愿 |
| **元数据并发抓取** | 1条/次 | 3条/次 | 10条/次 | 🔴CPU+网络IO | 配合队列，限制同时抓取数 |
| **元数据日抓取上限** | 30条 | 200条 | 无限制 | 🔴CPU+网络IO | 防止机器人刷接口 |
| **回收站保留天数** | 7天 | 30天 | 90天 | 🟡存储占用 | 影响数据安全感，不宜太短 |

#### 增值类（旗舰版专属或专业版部分开放）

| 维度 | 基础版 | 专业版 | 旗舰版 | 说明 |
|------|:--:|:--:|:--|:--|
| 批量操作 | 仅删除 | 删除+移动+标签 | 全部 | 批量操作合并请求，反而减少服务器压力，可作为营销点 |
| 导出格式 | HTML | HTML+CSV | HTML+CSV+PDF(未来) | 导出是用户信任基础，不应过度限制 |
| 优先客服响应 | ❌ | ❌ | ✅ | 纯服务层 |
| 早期功能尝鲜 | ❌ | ❌ | ✅ | 营销价值 |

| 维度 | 旧-基础 | 旧-专业 | 旧-旗舰 | **新-基础** | **新-专业** | **新-旗舰** | 变化逻辑 |

| 分组 | 30 | 100 | 200 | **∞** | **∞** | **∞** | 查询极轻，不限制 |
| 分享 | 20 | 50 | 200 | **∞** | **∞** | **∞** | 获客核心，改限容量 |

## 四、服务器资源预估（按新方案）

### 4.1 假设用户分布

参考 SaaS 行业典型转化率：

| 用户类型 | 占比 | 人均收藏 | 人均封面存储 | 日元数据抓取 |
|----------|:--:|:--:|:--:|:--:|
| 基础版 | 85% | 80条 | 5MB(仅用智能封面) | 5条 |
| 专业版 | 12% | 400条 | 50MB | 20条 |
| 旗舰版 | 3% | 2,000条 | 300MB | 50条 |

### 4.2 1,000 注册用户场景下的资源估算

| 资源项 | 旧方案估算 | **新方案估算** | 变化 |
|--------|:--|:--|:--|
| **PostgreSQL 数据** | ~300MB | ~400MB | 收藏上限取消，但 growth 更自然 |
| **COS 封面存储** | ~100GB | **~80GB** | 封面改容量限制，基础用户上传更少 |
| **Redis 元数据缓存** | ~200MB(24h TTL) | **~150MB** | 日抓取上限限制缓存膨胀 |
| **日均元数据抓取次数** | ~5,000次 | **~3,500次** | 日上限限制总请求数 |
| **share_items 表** | 增长不可控 | **可控** | 由单次分享容量严格限制 |
| **峰值 CPU 压力** | 高(封面处理无限制) | **中** | 元数据并发限制+封面容量限制 |

### 4.3 容量承载结论

新方案下，2核2GB 服务器的理论承载能力：

| 场景 | 安全DAU | 风险DAU |
|------|:--:|:--:|
| 以基础版为主(85%) | **150-200** | 400+ |
| 混合用户 | **120-180** | 350+ |
| 完成配额计数缓存优化后 | **400-600** | 1,000+ |

**关键改善**：
- 取消了 10,000 收藏的旗舰版 COUNT 查询压力（因为收藏不限但旗舰版用户占比仅3%）
- 元数据抓取日上限限制了外部请求总量
- 封面容量限制替代数量限制，更符合用户实际使用模式

### 5.2 行业趋势总结

1. **2024-2025 SaaS 趋势**：核心功能（数量类）趋向不限或给极宽松上限，限制转向**资源消耗型**维度（存储、AI调用、API频率）
2. **用户心理**："被限制数量" → 感觉产品小气；"被限制空间/频率" → 感觉合理（类似网盘逻辑）
3. **转化逻辑**：免费用户要能用爽，付费要让"重度用户"突破天花板，而非让轻度用户补齐基础功能

### 5.3 LinkChest 差异化定位

```
竞品共性：收藏数量是主要区分点
LinkChest 差异化：收藏无限制，限制"服务器真实成本"

营销话术：
- "你的收藏，我们管够——无限收藏、无限分组、无限标签"
- "付费只为更重的使用：更大封面空间、更快元数据抓取、更长回收站保留"
- "不拿数量卡脖子，你的数据你做主"
```

### 6.2 配额检查逻辑变更 (`quota.ts`)

| 变更项 | 旧逻辑 | 新逻辑 |
|--------|:--|:--|
| coverImages 计数 | 查 `coverImages` 表 COUNT | 查 `coverImages` 表 SUM(size)，换算为 MB |
| shareItems 计数 | 直接查 `shareItems` 表 COUNT | **删除此检查**，改在创建分享时校验 `maxItemsPerShare` |
| collections/tags/lists/shares | 硬上限检查 | **保留检查但上限设为 999,999**，防机器人滥用 |
| 新增元数据频率检查 | 无 | 新增 Redis 计数器，按 `metadataConcurrent` 和 `metadataDailyLimit` 限制 |

### 6.3 元数据抓取限流 (`metadata.ts`)

```typescript
// 新增：元数据抓取配额检查
async function checkMetadataQuota(userId: string, batchSize: number): Promise<boolean> {
  const { limits } = await getQuotaUsage(userId)
  
  // 1. 检查并发（正在进行中的抓取数）
  const concurrentKey = `lc:meta:concurrent:${userId}`
  const currentConcurrent = parseInt(await redis.get(concurrentKey) || '0')
  if (currentConcurrent + batchSize > limits.metadataConcurrent) return false
  
  // 2. 检查日上限
  const dailyKey = `lc:meta:daily:${userId}:${today}`
  const dailyUsed = parseInt(await redis.get(dailyKey) || '0')
  if (dailyUsed + batchSize > limits.metadataDailyLimit) return false
  
  return true
}
```

## 七、风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|:--:|:--:|:--|
| 收藏无限制导致单用户数据量暴增 | 低 | 中 | 设 100,000 防滥用上限；监控异常用户 |
| 封面容量限制技术实现复杂 | 中 | 低 | `coverImages` 表已有 size 字段，SUM 查询即可 |
| 元数据限流误伤正常用户 | 中 | 高 | 日上限 30 条对基础版足够；抓取失败不计数 |
| 旧用户已超限如何处理 | 中 | 中 |  grandfather 原则：已存在数据不删除，仅限制新增 |
| 分享无限导致 share_items 膨胀 | 低 | 高 | 单次分享容量严格限制，分享数量设 999,999 防滥用 |

## 八、总结

### 8.1 设计原则映射

| 原则 | 在新方案中的体现 |
|------|:--|
| **非服务器瓶颈不做限制** | 收藏/标签/分组/分享数量全部开放 |
| **影响服务器的均限制** | 封面容量、元数据频率、导入上限、回收站天数全部限制 |
| **客户体验强相关不做限制** | 收藏数量不限（体验核心），分享数量不限（获客核心） |
| **非体验强相关考虑限制** | 封面空间、元数据频率、导入量等体验边缘维度做套餐区分 |
| **按等级预估资源** | 基础版 85% 贡献极少资源，旗舰版 3% 贡献主要 COS/CPU 成本 |
| **参考其他产品** | 对标 Notion/Raindrop：核心不限，限增值资源 |

### 8.2 关键转变
