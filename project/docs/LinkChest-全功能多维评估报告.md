## 二、对服务器性能（CPU/内存）的影响评估

### 🔴 高影响（需要重点关注）

| 功能 | 影响点 | 评估详情 |
|------|--------|----------|
| **元数据抓取** (`metadata.ts`) | **CPU + 网络IO + 内存** | 添加收藏时，会级联发起 oEmbed → Bilibili API → OG-Scraper (HTML解析) → Cloudflare Worker Fallback 多个外部请求。HTML解析和JSON解析消耗CPU；多个请求同时挂起消耗内存和文件描述符。虽然有 **LRU内存缓存(500条)** 和 **Redis缓存(24h)**，但缓存未命中时对服务器冲击最大。 |
| **封面图片处理** (`cover.ts`) | **CPU密集型** | 使用 `sharp` 库进行 resize + WebP 编码，质量循环降级（每次减10）。上传高峰期会显著占用CPU。 |
| **批量导入** (CSV/HTML) | **CPU + DB压力** | 每批200条，创建大量记录。如果开启同步元数据抓取，会瞬间产生大量外部请求；当前已改为异步队列 (`metadata-queue`) 缓解。 |
| **配额查询** (`quota.ts`) | **DB查询压力** | 每次增删操作前触发，执行6次 `COUNT(*)`。虽有 **Redis缓存(30秒)**，但高并发下仍是热点。 |

### 🟡 中等影响

| 功能 | 影响点 | 评估详情 |
|------|--------|----------|
| **公开分享页面** (`public.ts`) | **CPU + 内存** | 缓存命中时极低（Redis直接返回）；缓存未命中时需查 `Share` + `ShareItem`（可能几十到几百条）并序列化。 |
| **回收站清理** (`scheduler.ts`) | **DB IO** | 每日凌晨批量物理删除，按100条分批处理，影响可控。 |
| **Next.js SSR** (`web`) | **CPU + 内存** | Web端页面服务端渲染，分享页面 `/s/[shareId]` 被大量访问时会消耗Node内存。 |

### 🟢 低影响

- **智能默认封面**：纯客户端SVG生成，**零服务器负担**（优秀设计）。
- **收藏列表查询**：纯DB分页查询，有索引覆盖，压力线性可控。
- **标签/分组管理**：单用户数据量小（最多100分组/100标签），查询极轻。

## 四、必要性评估（Must Have vs Nice to Have）

### P0：必须（产品存在的理由）

| 功能 | 必要性 | 理由 |
|------|--------|------|
| **收藏增删改查** | 🔴 必须 | 产品核心，没有就不叫收藏工具。 |
| **回收站（软删除）** | 🔴 必须 | 数据安全底线。用户误删是高频场景，物理删除会直接摧毁信任。 |
| **元数据自动抓取** | 🔴 必须 | "一键收入链藏"的核心价值。手动填标题封面等于退回书签时代。 |
| **配额系统** | 🔴 必须 | 商业化基础。没有配额就没有付费转化路径。 |
| **用户认证** | 🔴 必须 | 数据隔离和安全的基础。 |

### P1：高必要（核心组织能力）

| 功能 | 必要性 | 理由 |
|------|--------|------|
| **分组管理（含嵌套）** | 🟠 高必要 | 收藏的"文件夹"系统，300-10000条收藏没有分组等于无法使用。3级嵌套是合理折中。 |
| **标签管理** | 🟠 高必要 | 跨分组的第二维度组织，与分组形成"矩阵式"管理。 |
| **分享功能** | 🟠 高必要 | **获客核心**。用户分享→新用户导入→转化的闭环全靠它。10种类型覆盖足够全面。 |
| **导入导出** | 🟠 高必要 | 用户"带数据进来/带走数据"的权利，是信任基础。CSV+HTML双格式已完善。 |
| **封面上传/编辑** | 🟡 中高必要 | 个性化需求。但智能默认封面已覆盖80%场景，上传是补充。 |

### P2：中等必要

- **国际化**：已支持6种语言，覆盖主要市场。
- **深色模式**：现代Web应用标配。
- **统计模块**：满足用户"看看我有多少收藏"的心理需求。
- **Google登录**：降低注册门槛。

### P3：可选/待开发

- 浏览器扩展、收藏评论/笔记、评分、定期备份、邀请奖励细化。

## 五、对存储的压力评估

### PostgreSQL 数据库

| 数据表 | 存储压力 | 关键分析 |
|--------|----------|----------|
| **`share_items`** | 🔴 **最大** | 分享时**复制收藏快照**。用户创建1个含1000条收藏的"全部分享"，即新增1000条 `share_items`。数据随分享次数**成倍膨胀**。且目前表无自动清理机制（分享删除时级联删除）。 |
| **`collections`** | 🟠 大 | 核心表。super用户10000条 × 用户数。`deletedAt` 软删除意味着回收站数据也占空间（30天后清理）。`note` 字段如果未来支持长文本，压力会更大。 |
| **`error_events`** | 🟡 中 | 运维表。如果错误频发且不定期清理，会无限增长。当前有 `status` 字段和聚合计数，但未看到自动清理逻辑。 |
| **`share_views`** | 🟡 中 | UV统计，每用户每分享一条记录。如果分享 viral，数据量可观。有唯一索引 `[shareId, userId]` 控制上限。 |
| **`users`, `tags`, `lists`** | 🟢 小 | 单用户数据量有硬上限，整体可控。 |

### 腾讯云 COS 对象存储

| 存储项 | 压力 | 分析 |
|--------|------|------|
| **用户上传封面** | 🟠 中 | Sharp压缩为WebP，质量循环降级至目标大小。单张通常几十到几百KB。super用户最多3000张，总存量可控。 |
| **用户头像** | 🟢 极小 | 固定Key覆盖式上传，每人最多一张。 |
| **系统封面库** | 🟢 极小 | 预置有限数量，一次性。 |

### Redis 缓存

| 缓存类型 | 压力 | TTL | 分析 |
|----------|------|-----|------|
| 元数据 `md:{url}` | 🟡 中 | 24h | URL维度缓存，热门URL命中率高。但URL无限增长，Redis内存会持续累积（除非设上限）。 |
| 分享页 `share:{id}` | 🟡 中 | 7天 | 分享数据相对静态，缓存价值高。分享数量多后内存占用可观。 |
| 配额 `lc:quota:{id}` | 🟢 低 | 30s | 超短TTL，内存周转极快。 |
| 用户认证 `lv:user:{id}` | 🟢 低 | 300s | 活跃用户占用，量级小。 |
| 封面URL `cover:url:{id}` | 🟢 低 | 按过期时间 | 仅缓存字符串URL，极小。 |

**存储风险点**：`share_items` 是当前最隐蔽的存储炸弹。如果用户大量创建分享，数据库体积会快速膨胀。

## 六、对带宽的压力评估

### 🔴 高带宽消耗（需监控）

| 功能 | 带宽方向 | 分析 |
|------|----------|------|
| **公开分享页面访问** (`/s/[shareId]`) | **出向（服务器→用户）** | **最大带宽风险点**。分享链接可能被 viral 传播，大量匿名用户同时访问。Next.js SSR本身输出HTML + 分享数据JSON + 封面图加载（来自COS）。虽然有7天Redis缓存，但**首次访问和缓存过期后的并发**是瓶颈。 |
| **收藏列表API** (`GET /api/collections`) | 出向 | 高频API。每次返回多条收藏的标题、URL、封面图URL。分页限制了单次大小，但总流量随DAU线性增长。 |
| **封面图片加载** (COS URL) | 出向（COS→用户） | 不经过你的服务器，但**占COS流量费用**。如果分享页面被大量访问，封面图是流量大户。每个分享页可能加载几十张封面图。 |
| **封面上传** (`POST /api/upload/cover`) | 入向（用户→服务器） | Base64编码图片，体积比二进制大约33%。单次上传可能几MB。 |

### 🟡 中等带宽消耗

| 功能 | 分析 |
|------|------|
| **导入/导出** | 导出HTML/CSV时，一次请求可能返回较大体积（用户收藏越多越大）。 |
| **元数据抓取** | 入向：服务器请求外部平台。对服务器网卡出站带宽要求不高，但对**连接数和延迟**敏感。 |

### 🟢 低带宽消耗

- 标签/分组/认证API：数据量极小。
- 配额查询：有Redis缓存，且数据量小。
- 智能默认封面：SVG data URI，几KB级别，纯客户端生成。

## 九、关键建议与行动计划

### 立即关注（本周）

1. **分享模块的存储隐患**：`share_items` 表缺乏定期清理机制。如果分享被删除，级联删除会清理；但如果用户不删分享，数据永久保留。建议：
   - 为 `ShareItem` 添加 `createdAt` 索引，监控表大小增长。
   - 考虑对停用/过期的分享，在一段长时间后自动清理其 `share_items` 快照。

2. **元数据抓取的稳定性**：外部平台（尤其是小红书、抖音）频繁变更反爬策略。建议：
   - 监控 `fetchUrlMetadata` 的命中率/失败率。
   - Cloudflare Worker Fallback 是救命设计，确保 Worker 可用性。

### 短期优化（本月）

3. **虚拟滚动**：产品文档中已列为高优先级待开发（4h）。super用户10000条收藏在无虚拟滚动下体验会崩塌，建议尽快实施。

4. **ErrorEvent 表自动清理**：当前无清理逻辑。建议定时任务清理30天前的 `confirmed`/`ignored` 错误，防止无限增长。

### 中期规划（下季度）

5. **分享页带宽优化**：如果分享 viral，Next.js SSR + COS 封面图是双重流量消耗。可考虑：
   - 分享页纯静态化（ISR），配合增量更新。
   - 封面图增加CDN边缘缓存。

6. **PWA重新启用**：当前因 `@ducanh2912/next-pwa` 兼容性问题被注释掉。PWA对移动端Web体验提升显著，建议修复后上线。
