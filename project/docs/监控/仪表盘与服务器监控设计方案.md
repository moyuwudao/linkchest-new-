# LinkChest 仪表盘与服务器监控设计方案

> 本方案参考畅记项目（dang）的 MetricsService + AuditInterceptor 架构，
> 结合 LinkChest 的多环境（国内/海外）多服务器特性进行适配设计。

---

## 1. 畅记方案值得借鉴的核心设计

### 1.1 MetricsService 指标采集模式

| 设计点 | 畅记做法 | LinkChest 适配 |
|--------|----------|----------------|
| 实时指标 | Redis 计数器 + 定时聚合 | 复用 Redis，增加多服务器维度 |
| 每日汇总 | 定时任务写入数据库 | 复用现有定时任务框架 |
| 趋势数据 | 7日/30日滑动窗口 | 增加国内外对比维度 |
| 实时埋点 | 中间件自动采集 | 在 Express 中间件层采集 |

### 1.2 AuditInterceptor 审计模式

| 设计点 | 畅记做法 | LinkChest 适配 |
|--------|----------|----------------|
| 自动拦截 | NestJS Interceptor | Express 中间件 |
| 记录内容 | 操作人+操作类型+变更前后 | 增加服务器维度+环境变量变更 |
| 存储方式 | PostgreSQL 审计表 | 复用现有 PostgreSQL |
| 查询接口 | RESTful API | 新增 `/admin/audit-logs` 路由 |

## 2. LinkChest 仪表盘设计方案

### 2.1 核心指标体系

```
仪表盘指标架构
├── 1. 用户运营指标
│   ├── 注册用户数（日/周/月）
│   ├── 活跃用户数（DAU/WAU/MAU）
│   ├── 付费用户数
│   ├── 用户留存率（7日/30日）
│   └── 新增 vs 流失对比
│
├── 2. 内容运营指标
│   ├── 新建链接数（日/周/月）
│   ├── 活跃收藏夹数
│   ├── 分享链接数
│   ├── 收藏成功率
│   └── 内容分类分布
│
├── 3. 收入指标
│   ├── 总收入（USD/CNY）
│   ├── 订阅收入占比
│   ├── 付费转化率
│   ├── ARPU（每用户平均收入）
│   └── 退款率
│
├── 4. 平台性能指标
│   ├── API 响应时间（P50/P95/P99）
│   ├── API 成功率
│   ├── 错误率（按类型）
│   ├── PM2 进程状态
│   └── Redis 命中率
│
└── 5. 多环境对比指标
    ├── 海外 vs 国内用户比
    ├── 海外 vs 国内收入比
    ├── 各平台用户分布
    └── 各平台活跃度
```

### 2.2 数据可视化方案

| 指标类型 | 可视化方式 | 刷新频率 |
|----------|------------|----------|
| 核心数字卡片 | 大数字 + 环比箭头 | 5分钟 |
| 趋势图 | 折线图（7日/30日） | 5分钟 |
| 分布图 | 饼图/环形图 | 10分钟 |
| 排行榜 | 表格 + 进度条 | 10分钟 |
| 状态灯 | 绿/黄/红圆点 | 1分钟 |
| 对比图 | 双柱状图（国内外） | 5分钟 |

## 3. 服务器监控设计方案

### 3.1 监控指标体系

```
服务器监控架构
├── 1. 系统层监控
│   ├── CPU 使用率（%）
│   ├── 内存使用率（%）
│   ├── 磁盘使用率（%）
│   ├── 磁盘 I/O（MB/s）
│   ├── 网络 I/O（Mbps）
│   └── 系统负载（Load Average）
│
├── 2. 服务层监控
│   ├── PM2 进程状态
│   ├── PM2 内存使用
│   ├── PM2 重启次数
│   ├── Nginx 状态
│   ├── PostgreSQL 连接数
│   ├── Redis 内存/连接数
│   └── 服务响应时间
│
├── 3. 应用层监控
│   ├── API 接口响应时间（P50/P95/P99）
│   ├── API 错误率（4xx/5xx）
│   ├── 数据库查询耗时
│   ├── Redis 缓存命中率
│   ├── 慢查询日志
│   └── 错误日志统计
│
└── 4. 业务层监控
    ├── 登录成功率
    ├── 支付成功率
    ├── 收藏成功率
    ├── 分享成功率
    └── 各Provider可用性
```

### 3.2 数据采集方案

#### 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **A. 自研中间件** | 轻量、可控、无需额外组件 | 功能有限 | ⭐⭐⭐⭐ |
| **B. Prometheus + Grafana** | 功能强大、生态完善 | 部署复杂、学习成本高 | ⭐⭐⭐⭐⭐ |
| **C. 畅记模式（Redis+DB）** | 简单直接、与现有架构一致 | 实时性一般 | ⭐⭐⭐⭐ |

#### LinkChest 推荐方案：A + C 混合

```
数据采集流程
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ Express      │    │ 定时任务      │    │ 前端轮询      │
│ 中间件采集    │───▶│ 聚合写入      │───▶│ RESTful API  │
│ - 请求耗时    │    │ - 5分钟聚合   │    │ - 实时指标    │
│ - 错误计数    │    │ - 每日汇总    │    │ - 趋势数据    │
│ - 状态码     │    │              │    │ - 历史查询    │
└──────────────┘    └──────────────┘    └──────────────┘
        │                   │                   │
        ▼                   ▼                   ▼
   ┌─────────┐         ┌─────────┐         ┌─────────┐
   │ Redis   │────────▶│PostgreSQL│         │ 监控页面 │
   │ 计数器   │  定时同步│ 持久存储  │         │ 可视化   │
   └─────────┘         └─────────┘         └─────────┘
```

### 3.3 告警机制设计

| 告警级别 | 触发条件 | 通知方式 | 响应要求 |
|----------|----------|----------|----------|
| P0 - 紧急 | 服务不可用、数据库宕机 | 短信 + 邮件 | 5分钟内 |
| P1 - 重要 | API错误率>5%、响应时间>3s | 邮件 | 30分钟内 |
| P2 - 警告 | CPU>80%、内存>85% | 邮件 | 2小时内 |
| P3 - 提示 | 磁盘>70%、用户异常增长 | 系统通知 | 24小时内 |

## 4. LinkChest 现有架构集成方案

### 4.1 技术栈适配

| 组件 | 畅记技术 | LinkChest 技术 | 适配方案 |
|------|----------|----------------|----------|
| 后端框架 | NestJS | Express + TypeScript | 使用中间件模式替代Interceptor |
| 数据库 | PostgreSQL + TypeORM | PostgreSQL + Prisma | 新增 Metrics 和 AuditLog 表 |
| 缓存 | Redis | Redis | 复用现有Redis |
| 前端 | Next.js Pages | Next.js App Router | 使用 Server Components + Client 组件 |
| 部署 | 单服务器 | 多服务器 | 增加环境维度 |

### 4.2 数据库表设计

```sql
-- 指标采集表（Redis计数定时同步到此表）
CREATE TABLE metrics (
  id SERIAL PRIMARY KEY,
  timestamp TIMESTAMP NOT NULL,
  server VARCHAR(50) NOT NULL,      -- global/china-a/china-b
  metric_type VARCHAR(50) NOT NULL,  -- api_calls/errors/response_time/users
  metric_key VARCHAR(100) NOT NULL,  -- api.global.login/api.china.share
  value NUMERIC NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

-- 审计日志表
CREATE TABLE audit_logs (
  id SERIAL PRIMARY KEY,
  user_id INTEGER REFERENCES users(id),
  action VARCHAR(50) NOT NULL,       -- create/update/delete/configure
  entity VARCHAR(50) NOT NULL,       -- user/plan/config
  entity_id INTEGER,
  before_value JSONB,
  after_value JSONB,
  ip_address VARCHAR(50),
  server VARCHAR(50),
  created_at TIMESTAMP DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_metrics_timestamp ON metrics(timestamp);
CREATE INDEX idx_metrics_type_key ON metrics(metric_type, metric_key);
CREATE INDEX idx_audit_logs_user ON audit_logs(user_id);
CREATE INDEX idx_audit_logs_created ON audit_logs(created_at);
```

### 4.3 API 路由设计

```
/admin/metrics
├── GET /admin/metrics/realtime          # 实时指标
├── GET /admin/metrics/daily?date=xxx    # 每日指标
├── GET /admin/metrics/trend?days=7      # 趋势数据
└── GET /admin/metrics/compare           # 国内外对比

/admin/audit-logs
├── GET /admin/audit-logs                # 审计日志列表
├── GET /admin/audit-logs/:id            # 单条日志详情
└── GET /admin/audit-logs/export         # 导出CSV

/admin/servers
├── GET /admin/servers/status            # 服务器状态
├── GET /admin/servers/:id/health        # 单服务器健康检查
└── GET /admin/servers/alerts            # 告警列表
```

### 4.4 实施步骤

```
Phase 1: 数据采集层（1-2天）

└── 1.4 创建定时任务（Redis → DB 同步）

Phase 2: API 服务层（1-2天）
├── 2.1 创建 /admin/metrics 路由
├── 2.2 创建 /admin/audit-logs 路由
├── 2.3 创建 /admin/servers 路由
└── 2.4 添加管理员权限校验

Phase 3: 前端页面（3-5天）
├── 3.1 创建仪表盘页面（使用 ECharts/Recharts）
├── 3.2 创建服务器监控页面

└── 3.4 添加环境切换、刷新功能

Phase 4: 告警系统（1-2天）

```

## 5. 与畅记方案的核心差异

| 维度 | 畅记方案 | LinkChest 方案 |
|------|----------|----------------|
| 架构复杂度 | 单服务器 | 3台服务器（海外+国内A+B） |
| 数据采集 | NestJS Service | Express 中间件 |
| 数据聚合 | TypeORM + Redis | Prisma + Redis |
| 前端框架 | Next.js Pages | Next.js App Router |
| 监控维度 | 单一环境 | 多环境对比 |
| 部署方式 | 手动部署 | Git-Only 自动化部署 |
| 告警方式 | 系统内通知 | 邮件 + 系统通知 |
