## 2. 信息来源权威性标准

### 2.1 权威来源分级

| 级别 | 来源类型 | 可信度 | 使用场景 |
|------|----------|--------|----------|
| **L1（最高）** | 代码仓库中的配置文件、环境变量文件 | 100% | 生产环境配置确认 |
| **L2（高）** | 规则文档（.trae/rules/ 下的 .md 文件） | 95% | 日常查询、方案设计 |
| **L3（中）** | 部署脚本、构建脚本中的注释和配置 | 90% | 部署操作参考 |
| **L4（参考）** | 聊天记录、口头确认、临时笔记 | 70% | 需二次验证 |

### 2.2 信息验证机制

**验证流程：**

```
获取信息
    ↓
1. 【识别来源】确认信息来自 L1-L4 哪个级别
    ↓
2. 【交叉验证】L2/L3/L4 级别信息需与 L1 代码配置交叉验证
    ↓
3. 【时效检查】确认信息最后更新日期，检查是否过时
    ↓
4. 【一致性检查】对比多个来源，确认无矛盾
    ↓
5. 【标记可信度】标注信息级别和验证状态
```

**交叉验证规则：**

| 场景 | 验证方式 | 验证工具 |
|------|----------|----------|
| 服务器 IP | 对比 DEPLOY_RED_LINES.md 与 DEPLOYMENT.md | `grep` 搜索 |
| 环境变量值 | 对比 .env.example 与规则文档 | `Read` 工具 |
| Provider 配置 | 对比代码中 `lib/market.ts` 与规则文档 | `SearchCodebase` |
| 部署脚本参数 | 对比脚本内容与规则文档描述 | `Read` 工具 |

### 2.3 信息冲突处理

当不同来源信息冲突时：

1. **L1 代码配置优先** — 代码是事实标准
2. **规则文档以最新更新日期为准** — 较新的文档覆盖旧文档
3. **记录冲突** — 在信息查询结果中标注冲突点和处理依据
4. **触发文档更新** — 冲突说明文档已过时，需更新

## 6. 信息更新与维护机制

### 6.1 更新频率

| 信息类型 | 更新频率 | 责任人 | 触发条件 |
|----------|----------|--------|----------|
| 服务器 IP/端口 | 变更时 | 运维 | 服务器迁移、扩容 |
| 环境变量 | 变更时 | 开发 | 新增 Provider、密钥轮换 |
| Provider 配置 | 变更时 | 开发 | 新增/修改支付或登录渠道 |
| 部署经验 | 每次部署后 | 开发 | 部署遇到问题并解决 |
| 运营数据 | 每周 | 运营 | 定期数据汇总 |
| 合规要求 | 法规变更时 | 合规 | 法规更新 |

### 6.2 更新审批流程

```
发现信息需要更新
    ↓
1. 【确认变更】对比当前文档与实际配置，确认差异
    ↓
2. 【影响评估】评估变更影响范围（国内/海外/全局）
    ↓
3. 【审批确认】
   - 配置变更：需技术负责人确认
   - 部署信息：需运维确认
   - 运营方案：需产品/运营确认
    ↓
4. 【执行更新】修改对应规则文档
    ↓
5. 【验证更新】确保文档与代码配置一致
    ↓
6. 【通知相关人员】按第 8 节通知机制执行
```

### 6.3 版本控制方法

**文档版本号规则：**

```
版本格式：v主版本.次版本

- 主版本：重大架构变更、规则体系重构
```

## 7. 信息查询与使用权限管理

### 7.1 信息分级

| 级别 | 信息类型 | 访问权限 |
|------|----------|----------|
| **公开** | 功能开关、市场配置、通用架构 | 所有 Agent |
| **内部** | 服务器 IP、端口、目录结构 | 开发/运维 Agent |
| **敏感** | 数据库密码、API 密钥、JWT_SECRET | 仅授权 Agent |
| **机密** | 支付私钥、平台证书 | 禁止在对话中明文展示 |

### 7.2 查询规范

**查询基础信息时的标准流程：**

```
用户查询请求
    ↓
1. 【识别查询类型】运营/部署/配置/合规
    ↓
2. 【定位来源文档】
   - 运营 → MARKET-OPS.md
   - 部署 → DEPLOYMENT.md / DEPLOY_RED_LINES.md
   - 配置 → MARKET-OPS.md 第 4 节
   - 合规 → MARKET-OPS.md 第 7 节
    ↓
3. 【提取信息】使用 Read/SearchCodebase 工具读取文档
    ↓
4. 【验证时效】检查信息最后更新日期
    ↓
5. 【交叉验证】L2/L3 信息与 L1 代码配置对比
    ↓
6. 【输出结果】标注信息来源、级别、可信度
```

### 7.3 敏感信息处理

| 场景 | 处理方式 |
|------|----------|
| 查询数据库密码 | 告知配置位置，不展示具体值 |
| 查询 API 密钥 | 告知配置位置，不展示具体值 |
| 查询服务器 IP | 可直接展示（已在规则中公开） |
| 查询支付私钥 | 禁止展示，仅告知配置方式 |

## 8. 信息时效性检查流程

### 8.1 录入前时效检查

**在录入或查询基础信息前，必须执行以下检查：**

```
开始查询/录入
    ↓
1. 【检查文档日期】
   - 查看目标文档的最后更新日期
   - 如果超过 30 天未更新 → 标记"可能过时"
    ↓
2. 【检查关联代码】
   - 使用 SearchCodebase 搜索相关配置
   - 对比文档描述与代码实际配置
    ↓
3. 【检查部署记录】
   - 查看最近是否有部署变更
   - 检查是否有未记录的部署经验
    ↓
4. 【时效性判定】
   - ✅ 文档日期新 + 代码一致 → 信息有效
   - ⚠️ 文档日期旧 + 代码一致 → 信息有效但需更新日期
   - ❌ 文档与代码不一致 → 信息过时，需更新文档
```

### 8.2 时效性标记规范

查询结果中必须标注时效性状态：

| 标记 | 含义 | 操作 |
|------|------|------|
| ✅ 已验证 | 文档与代码一致，信息有效 | 直接使用 |
| ⚠️ 待确认 | 文档较旧或无法验证 | 建议二次确认 |
| ❌ 已过时 | 文档与代码不一致 | 禁止使用，需更新文档 |

### 8.3 自动时效检查触发点

以下场景自动触发时效性检查：

- 查询服务器 IP、端口等部署信息
- 查询环境变量配置
- 查询 Provider 启用状态
- 查询支付/登录渠道配置
- 查询数据库迁移状态
