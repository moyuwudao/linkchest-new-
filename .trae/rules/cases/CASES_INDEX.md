## 使用流程

```
遇到构建异常
    ↓
1. 识别异常类型（APK / 服务 / 环境）
    ↓
2. 查阅对应案例集锦
    ↓
3. 匹配错误关键词（搜索错误信息）
    ↓
4. 按案例解决方案执行
    ↓
5. 若未解决，记录为新案例并更新集锦
```

## 案例格式规范

每个案例遵循统一格式：

```yaml
---
id: CASE-001                    # 唯一编号
category: apk-build             # 分类标签
severity: high                  # 严重程度：critical/high/medium/low
frequency: frequent             # 发生频率：frequent/occasional/rare
first_seen: "2026-05-10"        # 首次发现日期
last_seen: "2026-05-14"         # 最近发现日期
status: resolved                # 状态：resolved/mitigating/ongoing
---
```

### 案例内容结构

| 章节 | 说明 |
|------|------|
| **现象** | 错误信息、日志片段、截图描述 |
| **根因** | 为什么会发生（技术原因） |
| **解决** | 具体解决步骤（可执行命令） |
| **预防** | 如何避免再次发生 |
| **相关** | 关联的其他案例编号 |

## 更新机制

### 何时更新案例集锦

| 场景 | 操作 |
|------|------|
| 遇到新的异常 | 新增案例，分配新编号 |
| 已知异常复发 | 更新 `last_seen` 和 `frequency` |
| 解决方案失效 | 更新解决步骤，保留历史记录 |
| 发现更优方案 | 补充替代方案，标记推荐 |

### 更新原则

1. **只增不改历史**：已记录的案例不删除，补充新信息
2. **频率驱动优先级**：高频案例优先完善预防方案
3. **规则引用案例**：规则文件通过链接引用案例，不重复内容
4. **定期回顾**：每月回顾一次案例集锦，合并相似案例

## 与规则文件的联动

### 规则文件如何引用案例

规则文件保持简洁，通过链接指向案例集锦：

```markdown
## 常见问题

详见 [APK构建异常案例集锦](cases/apk-build-errors.md)：
- Gradle 镜像失效 → 案例 CASE-001
- 缓存被 clean 删除 → 案例 CASE-002
- 图标被 prebuild 覆盖 → 案例 CASE-003
```

### 案例集锦如何补充规则

案例积累到一定数量后，提炼为规则：

```
案例 CASE-001 + CASE-002 + CASE-005 
    ↓ 提炼共同模式
新增规则："Gradle 镜像必须每周验证"
```

### 联动机制总结

| 层级 | 文件 | 职责 | 更新频率 |
|------|------|------|----------|
| **规则** | `BUILD.md`、`INTERACTION.md` | 定义"必须做什么" | 极少更新 |
| **案例** | `cases/*.md` | 记录"具体怎么解决" | 持续积累 |
| **索引** | `CASES_INDEX.md` | 汇总所有案例，快速检索 | 随案例更新 |

**工作流程：**

```
遇到异常
    ↓
规则文件："构建前必须检查缓存"（不变）
    ↓
案例集锦："CASE-002 缓存被删除的解决步骤"（持续补充）
    ↓
执行：按案例步骤解决问题
    ↓
记录：更新案例的 last_seen 和 frequency
```

**关键原则：**

1. **规则不动，案例流动**：规则保持稳定，案例持续积累
2. **先查案例，再执行**：遇到异常先搜索案例集锦
3. **案例驱动规则更新**：当多个案例揭示共同问题时，才更新规则
4. **避免规则膨胀**：规则只保留核心原则，细节放在案例中
